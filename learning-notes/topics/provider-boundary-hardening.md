# Граница совместимости провайдера в Bedmock

## Зачем она нужна

Bedmock принимает запрос в форме Bedrock, переводит его в каноническую модель и только затем
строит запрос к конкретному провайдеру. Опасное место — граница между канонической моделью и
transport: здесь параметр можно принять от пользователя, но незаметно потерять при переводе.

Хороший compatibility layer должен явно различать три ситуации:

1. параметр поддержан и действительно отображён в payload провайдера;
2. параметр не поддержан, поэтому запрос отклоняется до сетевого вызова;
3. пользователь осознанно разрешил совместимый passthrough, и библиотека предупреждает, что
   параметр был принят, но не передан провайдеру.

## Политика `fail` и `passthrough`

Политика задаётся для каждого provider profile:

```json
{
  "providers": {
    "openai": {
      "bedrock_controls": {"mode": "fail"}
    },
    "openrouter": {
      "bedrock_controls": {"mode": "passthrough"}
    }
  }
}
```

`fail` — безопасный режим по умолчанию. Неподдержанный Bedrock control вызывает
`ValidationException` до проверки API key и до сети. Это делает ошибку детерминированной и не
маскирует потерю параметра ошибкой авторизации.

`passthrough` сохраняет совместимость, но выдаёт один `BedmockCompatibilityWarning` на запрос со
списком отброшенных полей. Предупреждение — часть контракта: оно говорит, что Bedmock принял форму
запроса, а не что провайдер выполнил требование.

Transport объявляет реально отображаемые поля через `supported_bedrock_controls`. Поэтому
добавление поддержки нового control не требует менять общую политику: transport добавляет mapping
и объявляет соответствующее имя.

## Поток как конечный автомат

Потоковый ответ нельзя считать успешным только по HTTP 200. Ошибка может прийти первым SSE event
или после части токенов. Полезная модель состоит из двух состояний:

- **до первого канонического event** — retry безопасен, потому что вызывающий код ещё ничего не
  увидел;
- **после первого канонического event** — retry запрещён, иначе пользователь может получить
  дублированный текст или tool call.

Нормальное завершение требует terminal chunk с `finish_reason`. EOF без него и невалидный SSE —
`ModelStreamErrorException`, а не тихо завершившийся успешный поток. Ответ закрывается во всех
ветках, включая retry и исключения.

## Tool-result без потери соседних блоков

Сообщение может содержать `tool_result` вместе с текстом или изображением. При переводе в
OpenAI-compatible формат tool-result превращается в отдельное сообщение с ролью `tool`, а
оставшиеся блоки продолжают переводиться как исходное user-сообщение. Ранний `return` после
tool-result потерял бы соседний контент.

В streaming каждый text/tool block получает уникальный канонический индекс. Provider index у tool
call — это идентификатор внутри протокола провайдера, а не готовый Bedrock content-block index;
эти пространства индексов нельзя смешивать.

## Что проверяют тесты

- `fail` останавливает запрос до transport;
- `passthrough` выдаёт ровно одно предупреждение и сохраняет controls в каноническом запросе;
- поддержанный control не предупреждает;
- tool-result не удаляет текст и изображение;
- retry происходит только до первого выданного event;
- provider error, оборванный поток и malformed SSE становятся типизированными Bedrock-ошибками;
- несколько tool calls и текст получают разные content-block indexes;
- malformed успешный HTTP-ответ провайдера классифицируется как внутренняя ошибка, а не ошибка
  пользователя.
